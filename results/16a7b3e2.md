## Outcome

排查了子代理超时后报告生成路径上的两处明显 bug，这些 bug 会导致子代理产出的最终报告被损坏或被写入错误路径，使得父进程收到的报告永远达不到 complete 标准。

### Bug 1: `subagent-improve` 模板中 `resultFilePath` 被错误替换为 `\n`

在 `executeSubagentImprove` 的 agentMessage 模板中，原文为：
```
**Write your final report to the result file, not in the chat.** Persist your complete report to this path using the write tool first:
\nThen write a brief summary to the chat.
```
这里的 `\n` 是一个字面反斜杠+n，而不是模板变量 `${resultFilePath}\n`。这意味着子代理被告知将报告写到 `\nThen write a brief summary...`（根本不是一个有效路径），导致最终报告永远不会落到预期的 `resultFilePath`。父进程在 `waitForFinalTaskReport` 中轮询永远等不到文件，最终 fallback 到 "Subagent did not finish within the timeout" 的 partial 报告。

**注意**：`run-agent-task` 的模板已经是正确的 `${resultFilePath}\n`，但 `subagent-improve` 分支复制时漏掉了 `${resultFilePath}`。

### Bug 2: `resolveSubagentFinalReport` 中 fallback 报告被拼接损坏

在 `subResult.success && output` 分支中，原始代码：
```
return `Status: partial\n\n## ...\n${output.slice(0, 800)}\nStatus: partial\n\n## Outcome\nSubagent did not finish within the timeout.\n...`；
```
这里将两个不同的 partial 报告模板拼接在了一起（`Captured output` + `Subagent did not finish`），导致最终报告包含重复的 `## Outcome`、`## Changes` 等段落。

同时失败分支中 `Status: paStatus: failed` 也是拼写错误，多了一个 `pa` 前缀。

这两个 bug 叠加后的效果是：子代理即使成功写完了报告文件，但 resolveSubagentFinalReport 在轮询到该文件前，可能先走到这个损坏的 fallback 路径，产生一个既包含 partial 内容又包含重复段落的报告，导致 `isFinalTaskReport()` 检查失败，最终父进程认为子代理超时。
